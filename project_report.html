—Abstract

This project presents a fully functional keypad-based smart door lock built on the Arduino Uno microcontroller. A 4×4 matrix keypad serves as the PIN entry interface; a servo motor acts as the physical locking mechanism; and an LED provides real-time status feedback. Passwords are stored in the microcontroller's non-volatile EEPROM, persisting across power cycles. The system includes a 3-attempt lockout mechanism and a secure password-change workflow. Three firmware bugs identified during development were resolved — covering EEPROM validation, special-key filtering, and input buffer overflow. The project demonstrates practical competency in embedded C++ programming, digital I/O, PWM control, and hardware debugging.

01Introduction

Traditional mechanical door locks depend entirely on physical keys, which are susceptible to being lost, copied, or forgotten. Electronic lock systems offer a more adaptable and auditable alternative — enabling PIN-based access, remote control capabilities, and event logging that physical keys cannot provide.

This project implements a complete electronic door lock using low-cost, widely available components, targeting the Arduino Uno as the control unit. It was developed as part of an internship training programme in Electronics and Communication Engineering, with the objective of demonstrating applied understanding of microcontroller interfacing, non-volatile memory management, and event-driven firmware architecture.

Objectives

The system was designed to meet the following goals:

  • Accept a 4-digit PIN via physical keypad
  • Control a servo motor as the door lock actuator
  • Persist the password across power cycles using EEPROM
  • Provide a secure password-change workflow
  • Implement a brute-force lockout mechanism

03System Design

Architecture Overview

The firmware follows an event-driven polling architecture. The main loop() function continuously polls the keypad for input. When a key is detected, it is routed to one of three handlers based on the key value — clear (*), submit (#), or change-password (A). Digit keys are appended to an input buffer. Output — servo position and LED state — is triggered by the result of the password comparison.

EEPROM Password Storage

The 4-digit password is stored as four ASCII character bytes at EEPROM addresses 0–3. On every startup, loadPassword() reads these four bytes and validates that each falls within the ASCII digit range (decimal 48–57, i.e., '0'–'9'). If any byte fails validation — as would occur on a factory-fresh board where EEPROM initialises to 0xFF — the system sets the default password to 1234 and writes it to EEPROM before proceeding.

04System Operation

Normal Unlock Flow

On power-up, the system loads the stored PIN and initialises the servo to 0° (locked). The user enters up to four digits using the keypad, then presses # to submit. If the input matches the stored password, the servo rotates to 90° and the LED activates. After a fixed 5-second window, the servo automatically returns to 0° and the LED turns off — relocking the door without any further user input.

Wrong Attempt & Lockout

Each incorrect PIN increments a wrongAttempts counter. Upon reaching three consecutive failures, the system enters a 30-second lockout state during which all keypad input is silently ignored. Once the lockout expires, the counter resets to zero and normal operation resumes. This mechanism prevents automated brute-force PIN guessing.

Password Change

Pressing A initiates the password-change workflow. The system first requires the current PIN to be entered correctly — providing authentication before any change is permitted. Only after successful verification does the system prompt for and accept a new 4-digit PIN. The new password is immediately written to EEPROM and becomes active for all subsequent unlock attempts. Crucially, only digit keys (0–9) are accepted during this flow; special keys are filtered out to prevent an unenterable password from being set.

void checkPassword() {
  if (input == password) {
    Serial.println(" ACCESS GRANTED");
    digitalWrite(LED_PIN, HIGH);
    lockServo.write(90);    // unlock
    delay(5000);
    lockServo.write(0);     // auto-relock

      delay(30000);         // 30s lockout
      wrongAttempts = 0;
    }
  }
  input = "";
}

06Bugs Identified & Resolved

During development and code review, three functional bugs were identified and fixed before final submission.

Bug 01 EEPROM Blank Board — Incorrect Password Loaded

Root Cause: On a factory-fresh or erased Arduino, all EEPROM bytes default to 0xFF (decimal 255). The original validation checked password.length() != 4, which always returned false because four 0xFF bytes do form a 4-character string — just an unprintable, unenterable one. The system would then operate with a garbage password that no user input could ever match.

Resolution

Each EEPROM byte is now explicitly validated to be within the ASCII digit range (48–57). Any byte outside this range causes the system to discard the stored value and set the default password to 1234.

Bug 02 Special Keys Accepted During Password Change

Root Cause: The password-change routine accepted all keypad input indiscriminately, including *, #, and letter keys (A–D). This allowed a user to set a password containing non-digit characters — a password that could never be re-entered through the normal unlock flow, permanently locking out the system.

Resolution

A dedicated collectCode() helper was introduced. It only appends a key to the buffer if the key falls in the range '0'–'9', silently ignoring all other keys during both old-password verification and new-password entry.

Bug 03 Input Buffer Overflow — Comparison Always Fails

Root Cause: The main loop() had no upper bound on the input string length. A user who typed more than four digits before pressing # would generate a string longer than the stored 4-character password. Since Arduino's String comparison is exact, a 5+ character input could never equal a 4-character password, causing the unlock to fail silently regardless of the digits entered.

Resolution

A guard condition input.length() < PASS_LEN was added before any digit is appended to the input buffer, capping it at exactly 4 characters.

08Conclusion

This project successfully delivers a working embedded door lock system that integrates keypad input, servo actuation, LED feedback, and persistent EEPROM storage into a cohesive security application. The firmware was iteratively developed and debugged — with three real bugs identified, root-caused, and resolved — demonstrating a methodical approach to embedded software quality.

The system is intentionally scoped for clarity and executability within a standard lab setup, while remaining architecturally extensible. The SoftwareSerial bridge to an optional ESP8266 module would allow future enhancement with WiFi-based remote unlock and cloud access logging via a service such as Adafruit IO, without requiring changes to the core Uno firmware.
